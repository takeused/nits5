# 기술 트렌드 분석 검토 보고

검토일: 2026-07-13
대상: `js/chart.js`의 `runTrendAnalysis → fetchTrendData → calcTrendPhase → renderTrendDashboard` 흐름

## 결론

현재 화면은 결과 카드와 차트가 짙은 네이비·블루 기반으로 고정되어, 홈·검색·연구비 화면의 graphite/slate 톤과 시각적으로 분리된다. 기능 측면에서는 단순 오류보다 **표본이 전체 분포를 대표한다고 가정하는 부분**이 가장 큰 위험이다. 특히 최근성·관련성 순으로 반환될 수 있는 API 페이지를 연도 분포 표본으로 사용한 뒤 전체 건수 비율로 확대하는 방식은 연도별 추세를 왜곡할 수 있다.

## 우선순위별 발견

### P0 — 연도별 추세 표본 편향 가능성

- 근거: `fetchYearDist()`가 대상별 최대 5페이지(페이지당 100건)만 수집한 뒤 `total / fetched` 비율로 각 연도 건수를 일괄 확대한다.
- 문제: ScienceON 검색 결과의 페이지 순서가 연도 균등 표본이라는 보장이 없다. 관련도순·최신순·기관별 편향이 있으면 특정 연도가 과대/과소 추정된다.
- 영향: “성장/급성장/정체” 단계와 최근 3년 집중도, 논문·특허 비율이 모두 왜곡될 수 있다.
- 권고: 연도 필터 API가 있으면 연도별 직접 집계로 전환한다. 없으면 표본 기반 추정임을 수치와 함께 표시하고, 일괄 비율 확대를 제거하거나 불확실성 구간을 제공한다.

### P0 — 전체 건수와 연도 분류 건수의 분모 불일치

- 근거: `total`은 전체 레코드 수지만, 연도 파싱 실패 레코드는 `counts`에서 제외된다. 그럼에도 `total / fetched`를 그대로 적용한다.
- 영향: 연도 미상 레코드가 많을수록 연도별 합계가 실제 전체보다 부풀려질 수 있다.
- 권고: `yearKnownCount`, `yearUnknownCount`, `sampleCoverage`를 별도 산출하고, 연도 미상 비중이 높으면 추세 단계 산출을 보류하거나 신뢰도 하향한다.

### P1 — 분석 기간이 고정되어 현재 시점과 어긋남

- 근거: `startYear = 2017`, `endYear = 2025`가 하드코딩되어 있다. 현재가 2026년인데 로딩·설명 모두 2017~2025로 고정된다.
- 영향: 최신 연도 누락, 사용자가 보는 “최근 3년” 정의와 실제 데이터 범위의 혼동이 발생한다.
- 권고: 기준일에서 `endYear`를 계산하고, API 최신 색인 지연을 고려해 “최신 완결 연도”를 명시한다. 기간은 설정값으로 분리한다.

### P1 — 0건·초저건 분포에서 단계/차트가 불안정

- 근거: `calcTrendPhase()`는 이전 3년 평균이 0이면 `rate` 없이 “유망 분야”를 반환한다. 모든 값이 0이면 최고 연도를 배열 첫 연도로 선택한다. 차트 `suggestedMax`도 최대값이 0이면 0이 된다.
- 영향: 데이터가 없는데 유망하다고 표시되거나, 2017년이 피크로 표시되는 등 잘못된 해석이 가능하다.
- 권고: `total === 0`, `knownYearCount < 최소표본`, `max === 0`을 별도 상태로 처리하고 단계·피크·비율 인사이트를 생성하지 않는다. 차트에는 “데이터 부족” 상태를 표시한다.

### P1 — HTTP/API 오류가 부분 성공으로 조용히 축소됨

- 근거: `fetchYearDist()`는 `resp.ok`를 일반적으로 확인하지 않고, XML 파싱·API 오류가 발생하면 현재까지의 빈약한 결과를 반환한다. 429 재시도도 2회 후 구분된 실패 상태를 남기지 않는다.
- 영향: API 오류를 실제 0건으로 오인할 수 있고, 논문과 특허 중 한쪽만 실패해도 비교 지표가 생성된다.
- 권고: 각 대상에 `status`, `httpStatus`, `errorCode`, `fetched`, `total`을 반환하고, 한쪽 실패 시 결과를 “부분 분석”으로 표시한다. 실제 0건과 오류 0건을 UI에서 분리한다.

### P1 — 트렌드 단계가 표본 신뢰도를 반영하지 않음

- 근거: 최근 3년 평균과 직전 3년 평균의 단순 증감률만으로 단계 임계값(+30/+10/-10)을 판정한다.
- 영향: 표본 수가 작거나 추정 확대가 적용된 경우 작은 절대 변화가 큰 비율로 보일 수 있다.
- 권고: 최소 건수·연도 완결성·표본 커버리지·추정 여부를 단계 판정에 반영하고, 단계와 별도로 데이터 신뢰도 등급을 표시한다.

### P2 — 지표 의미와 라벨이 혼동될 여지

- 근거: 특허÷논문×100을 “연구–IP 전환 공백 신호”와 연계해 설명하지만, 이는 특허·논문 색인 규모와 검색어 민감도에 크게 좌우된다.
- 영향: 사용자가 비율을 사업화 가능성 또는 실제 전환율로 오해할 수 있다.
- 권고: “색인 비율 기반 탐색 신호”로 명칭을 낮추고, 성공확률·전환율이 아님을 KPI 옆에 상시 표시한다. 논문·특허 검색 범위와 API 상태를 함께 노출한다.

### P2 — 요청 흐름이 순차적이고 중복 호출이 있음

- 근거: 토큰 검증 검색 1회 후 논문 수집, 500ms 대기, 특허 수집을 순차 실행한다.
- 영향: 체감 대기시간이 길고, 토큰 검증과 실제 검색이 중복된다.
- 권고: 토큰 상태를 캐시하고, 검증이 필요한 경우에만 수행한다. API rate limit을 고려한 제한적 병렬화와 취소(AbortController)를 도입한다.

## 디자인 검토

- 현재 `analysis-header`와 내부 추세 차트가 `#0f172a~#1e293b`, 파란·주황 고채도 팔레트를 인라인으로 고정한다.
- 앱 공통 graphite 토큰(`--graphite`, `--graphite-hover`)과 표·카드의 중성 표면을 사용하지 않아 연구비·검색결과 화면과 이질적이다.
- 인라인 스타일이 많아 다크모드·브랜드 톤 변경 시 일괄 조정이 어렵다.
- 권고 순서: (1) 분석 카드/차트 컨테이너를 공통 surface 토큰으로 이동, (2) 차트 색을 graphite + muted blue/orange로 낮춤, (3) 상태·오류·신뢰도 배지를 공통 컴포넌트화, (4) 인라인 스타일을 CSS 클래스로 이동.
